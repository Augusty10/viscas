# Gmail User Flow

## Overview

This document describes how users interact with the Gmail module and how requests flow through the Viscas system.

Every Gmail action follows a secure server-side architecture where the frontend never communicates directly with Google APIs.

# API Mapping

| User Action | API | Service |
|-------------|----------------------------|----------------|
| Load Inbox | GET /api/v1/gmail/inbox | Gmail API |
| Open Email | GET /api/v1/gmail/message/:id | Gmail API |
| Send Email | POST /api/v1/gmail/send | Gmail API |
| Search Email | GET /api/v1/gmail/search | Gmail API |
| AI Summary | POST /api/v1/ai/email-summary | Corsair AI |
| Smart Reply | POST /api/v1/ai/smart-reply | Corsair AI |
| Archive Email | PATCH /api/v1/gmail/archive | Gmail API |
| Delete Email | DELETE /api/v1/gmail/delete | Gmail API |

# Security

- OAuth Required
- Backend-only Gmail access
- Token validation on every request
- HTTPS communication
- Request validation
- Rate limiting

- Infinite scrolling

# Summary

The Gmail User Flow is designed around a secure, server-side architecture. Every user interaction passes through the Viscas backend before reaching Gmail or Corsair AI, ensuring centralized authentication, consistent business logic, enhanced security, and a seamless user experience.
